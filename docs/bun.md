# [bun](https://bun.sh/)

## 常用命令

```sh
bun create
bun install
bun run
bun test
bun dev
bun build
bun --hot
```


## 从Node.js迁移到Deno或Bun，分别会遇到哪些主要的技术挑战和迁移成本？

将一个现有的 Node.js 项目迁移到 Deno 或 Bun 是一项重要的技术决策。为了帮助你清晰地了解其中的挑战和成本，下面这个表格从多个维度对比了迁移到这两个运行时的主要差异。

| 挑战维度 | 迁移到 Deno | 迁移到 Bun |
| :--- | :--- | :--- |
| **模块系统** | **挑战巨大**。需将 CommonJS 改为 ES 模块，可能需修改 `require`/`module.exports` 为 `import`/`export`，并处理特定模块加载逻辑 。 | **挑战较小**。高度兼容 CommonJS 和 ES 模块，通常无需修改代码 。 |
| **包管理与依赖** | **挑战巨大**。需改变依赖引入方式（使用 URL 或 `deps.ts`），并替换不兼容的 npm 包 。 | **挑战很小**。设计上高度兼容 npm 和 `package.json`，多数情况下运行 `bun install` 即可 。 |
| **安全性模型** | **需要适应**。默认安全沙箱环境，需为脚本显式授权（如 `--allow-net`），可能需调整开发和部署流程 。 | **近乎无感**。继承 Node.js 的开放权限模型，通常无需额外配置 。 |
| **原生模块与 API 兼容性** | **挑战显著**。依赖 Node.js 原生模块（C++ 插件）或特定底层 API 的库可能无法工作，需寻找替代方案或用 Deno 的 FFI 重写 。 | **挑战中等**。多数 Node.js API 兼容，但部分原生模块可能需要重新编译，某些特定 API（如 `child_process.fork`）可能支持不完整 。 |
| **工具链与配置** | **需要调整**。拥有内置工具链（格式化、测试等），可替换部分现有工具，但需学习新命令和工作流 。 | **简化整合**。提供一体化工具链（包管理、构建、测试），可简化配置，但需评估与现有 CI/CD 流程的整合 。 |
| **生态系统成熟度** | 相对年轻，第三方库和框架选择较少，企业级运维工具支持可能有限 。 | 较新，但发展迅速。兼容 npm 生态是其巨大优势，但某些专业或深度依赖 Node.js 特性的工具可能支持不足 。 |

### 💡 深入理解迁移挑战

除了表格中的对比，还有一些更深层的挑战需要考虑：

*   **迁移到 Deno 的额外挑战**
    *   **工具链替换**：Deno 提供了一套内置工具，如代码格式化 (`deno fmt`)、 linting (`deno lint`) 和测试运行器 (`deno test`)。虽然这可以减少项目对 Prettier、ESLint 等工具的依赖，但也意味着团队需要学习新的命令和工作流程，并可能需要对现有配置进行迁移 。
    *   **架构差异**：Deno 的模块系统去中心化，直接通过 URL 引入模块。这虽然灵活，但也带来了版本管理和更新控制的挑战，通常需要借助 `deps.ts` 文件来集中管理依赖 。

*   **迁移到 Bun 的性能权衡**
    *   Bun 基于 Zig 语言和 JavaScriptCore 引擎，其架构设计与基于 V8 的 Node.js 不同。有分析指出，Bun 在**冷启动速度**上优势明显（例如在 Serverless 场景下），但在长时间运行、高计算负载的任务中，Node.js 的 V8 引擎经过深度优化后可能表现出更稳定的性能 。这意味着需要根据应用的具体负载类型评估性能收益。

### 🚧 评估迁移成本

迁移成本涉及时间、人力和资源投入，主要包括：

*   **代码修改量**：这是最直接的成本。迁移到 Deno 通常意味着大范围的代码重写，特别是模块引入方式。而迁移到 Bun，如果项目规范且未使用生僻的 Node.js 特性，代码修改量可能很小 。
*   **依赖项评估与测试**：需要逐一检查项目依赖的 npm 包在目标运行时中的兼容性。对于 Deno，可能需要寻找替代库；对于 Bun，需测试其实际运行情况。这部分测试和适配工作会占用大量时间 。
*   **团队学习成本**：团队需要学习 Deno 或 Bun 的新特性、API 和最佳实践。Deno 的安全模型和模块系统概念较新，学习曲线可能更陡峭 。
*   **基础设施与工具链调整**：包括更新 CI/CD 流水线、Docker 镜像、部署脚本等，以适应新的运行时和工具命令 。

### 💎 如何决策：Deno、Bun 还是坚守 Node.js？

面对迁移挑战，你的决策应基于项目的具体情况：

*   **考虑迁移到 Deno，如果**：
    *   项目是**全新的**，并且**安全性是首要考量** 。
    *   团队希望**拥抱最新的 Web 标准**，并愿意接受不同的开发模式 。
    *   项目规模不大，或依赖较少，易于重构。

*   **考虑迁移到 Bun，如果**：
    *   首要目标是**提升现有 Node.js 应用的性能和开发效率** 。
    *   希望用**一体化工具链简化技术栈**（用 Bun 替代 npm、Webpack、Jest 等） 。
    *   项目需要良好的 Node.js 生态兼容性，同时能接受一个快速演进但相对年轻的技术 。

*   **暂时保持使用 Node.js，如果**：
    *   项目是**大型、复杂且成熟的企业级应用**，稳定性至关重要 。
    *   项目**深度依赖大量特定的 Node.js 原生模块或 API**，且找不到合适的替代方案 。
    *   团队**没有足够的资源**来应对迁移带来的重构、测试和风险控制。

最终，是否迁移以及选择哪个运行时，取决于你在性能、开发体验、安全性方面的需求，与所需投入的迁移成本、潜在风险之间的权衡。**对于现有大型项目，更稳妥的做法通常是先在一个小的新项目中尝试 Deno 或 Bun，或者在现有项目的非核心工具脚本中逐步引入，积累经验后再做全局决策。**

希望这些分析能帮助你做出更明智的决策。如果你能分享更多关于你项目类型和迁移目标的信息，或许我可以提供更具体的建议。


## Bun对Node.js原生模块的兼容性具体支持到什么程度？有哪些常见的兼容性问题需要特别注意？

Bun 在设计目标上致力于成为 Node.js 的完全兼容替代品，并且已经取得了显著进展，特别是从 1.2 版本开始，其兼容性得到了大幅提升。不过，在实际使用中，确实存在一些需要特别注意的兼容性差异。

下表为你梳理了主要 Node.js 原生模块在 Bun 中的兼容性概况，可以帮你快速建立一个整体印象。

| 模块名称 | Bun 中的支持状态 | 关键说明 |
| :--- | :--- | :--- |
| **`node:fs`** (文件系统) | 🟡 大部分支持 | 核心功能稳定，但部分进阶 API（如 `fs.fdatasyncSync`）可能未实现或行为有差异。 |
| **`node:http` / `node:https`** | 🟢 完全实现 | HTTP 1.x 服务稳定支持，Express 框架性能在 Bun 1.2 中有显著提升。 |
| **`node:path`**, **`node:os`** 等工具模块 | 🟢 完全实现 | 可安全使用。 |
| **`node:http2`** | 🟡 客户端/服务器支持 | Bun 1.2 版本起提供了兼容性实现，但需留意与原生 Node.js 实现的细微差别。 |
| **`node:child_process`** (子进程) | 🟡 部分支持 | `spawn` 等基本功能可用，但 `fork()` 的 IPC 通信等功能可能不完整或仅在与 Bun 进程间有效。 |
| **`node:cluster`** (集群) | 🟡 基本支持 | Bun 1.2 开始支持多进程应用模式。 |
| **`node:zlib`** (压缩) | 🟡 支持并增强 | Bun 1.2 中使用原生代码重写，性能提升并新增 Brotli 算法支持。 |
| **`node:dgram`** (UDP) | 🟡 基本支持 | 支持创建 UDP Socket，使依赖此模块的包（如 `dd-trace`）可在 Bun 中运行。 |
| **`node:crypto`** (加密) | 🟡 大部分支持 | 常用哈希、加密算法稳定，但一些进阶功能（如 `crypto.Certificate`）可能缺失。 |
| **`node:worker_threads`** (工作线程) | 🟡 部分支持 | Worker 基本功能可用，但部分选项（如 `resourceLimits`）和支持特定 Node.js 特性的 API 可能缺失。 |

### 💡 常见的兼容性问题与应对策略

了解状态后，我们来看看在实际项目中更容易遇到的具体问题及其解决办法。

1.  **原生模块（C++ 插件）的兼容性**
    -   **问题核心**：这是最具挑战性的领域。那些通过 `node-gyp` 编译的、依赖 Node.js V8 引擎特定 C++ API 的原生模块（如某些数据库驱动、加密库），在基于 Zig 和 JavaScriptCore 的 Bun 中可能无法直接运行。
    -   **应对策略**：
        -   **寻找纯 JavaScript 替代**：优先考虑使用功能相同的纯 JavaScript 实现库。例如，用 `bcryptjs` 替代 `bcrypt`。
        -   **检查官方兼容性列表**：在尝试使用一个依赖原生模块的 npm 包前，先查阅 Bun 官方文档或相关社区讨论，确认其兼容性。
        -   **关注 V8 API 兼容层**：Bun 团队在 JavaScriptCore 上实现了一层 V8 的公共 C++ API 兼容层，旨在使部分依赖 V8 API 的包（如 `cpu-features`）能够运行，但这仍处于持续改进阶段。

2.  **ESM 与 CommonJS 的模块互操作**
    -   **问题核心**：虽然 Bun 同时支持 ESM（`import`）和 CommonJS（`require`），但在混合使用时可能遇到问题。例如，在 ESM 文件中尝试用 `require` 动态导入一个 CommonJS 模块，可能会因为 Bun 默认的模块解析规则而报错。
    -   **应对策略**：
        -   **统一模块规范**：在 `package.json` 中明确设置 `"type": "module"`，并尽量使用 ESM 的 `import` 语法。
        -   **使用动态导入**：如需条件导入，使用 ESM 的 `await import('./module.cjs')` 来替代 `require`。

3.  **API 行为的细微差异**
    -   **问题核心**：即使同一个 API，Bun 和 Node.js 的实现也可能存在微小差异。例如，`fs.promises.readFile` 在某些情况下返回的可能是 `Uint8Array` 而非 Node.js 的 `Buffer`（尽管 Bun 中的 `Buffer` 全局对象是可用的）。某些流（Stream）处理 API 的错误处理或背压控制逻辑也可能不同。
    -   **应对策略**：
        -   **善用 Bun 原生 API**：对于文件操作，Bun 提供了更现代、一致的 `Bun.file()` API，可以规避此类问题。
        -   **全面测试**：对文件 I/O、网络请求等关键模块进行充分的测试，确保其在 Bun 下的行为符合预期。

4.  **环境变量加载**
    -   **便利性与差异**：Bun 的一个优点是会自动加载项目根目录下的 `.env` 文件，无需像 Node.js 那样手动引入 `dotenv` 包。但这也意味着代码中手动调用 `dotenv.config()` 的地方可能变得多余，在混合环境中需要注意。

### 💎 迁移与开发实践建议

为了平稳地使用 Bun，这里有一些实践建议：

-   **渐进式迁移**：对于现有 Node.js 项目，不要急于全部切换。可以先用 `bun install` 替代 npm/yarn 来管理依赖，感受速度提升。然后尝试用 `bun run` 来运行脚本或启动应用，逐步验证兼容性。
-   **利用 Bun 的优势**：对于新项目，可以积极尝试 Bun 的原生高端 API，如高性能的 `Bun.serve` 创建 HTTP 服务器、`Bun.sql` 连接数据库，以及内置的测试框架，这些通常能带来更好的开发和运行体验。
-   **保持更新与查阅文档**：Bun 正在快速迭代，兼容性问题在不断被修复。保持 Bun 版本为最新，并遇到问题时优先查阅 https://bun.sh/docs，其中有一个页面专门记录了与 Node.js 的兼容性状态。
